计算机网络

计算机网络，作为一个前端人er

计算机网络为什么那么重要，在各大程序员交流论坛去看前端必备技能，已经从三大件渐渐过渡到完整的前端知识体系中。而计算机网络变成为一个前端从业人员必备的技能之一。

对计算机网络中个人也存在很多模凌两可的认知，希望这篇文章能够对自己和学习前端的同行有些许的帮助。

应用层

计算机网络：客户端想要和服务端交换资源，便需要通过网络作为载体。

计算机网络逐渐完善并规范了两种之间的传输规范和约束。如计算机网络中的OSI七层模型。

应用层：定义了应用进程间的交互和通信的规则，不同主机如何互相传递报文，传递报文的格式、类型以及一些字段等。

HTTP协议

HTTP协议：HTTP超文本传输协议，规定了客户端和服务器之间传递报文的格式和方式，默认80端口。使用TCP作文传输层协议，保证了传输的可靠性。

HTTP是一个无状态的协议，服务端不会保存客户端的任何信息。

HTTP有两种连接模式：非持续连接和持续连接。

  • 非持续连接：服务器需要为每一个请求对象建立和维护一个新的连接。
  • 持续连接：服务器回复用连接，TCP连接不关闭，多个请求复用一个TCP连接。

持续连接的好处：采用持续连接可以避免每次建立TCP连接所花费的时间。

HTTP1.0：使用持续连接，可以通过使用connection：keep-alive要求服务器不关闭TCP连接

HTTP1.1：默认开启持续连接，目前对于同一个域而言，通常可以建立最多6个TCP持久连接。

HTTP请求报文

对于HTTP报文而言，有两种报文：请求报文和响应报文。

请求报文由：请求行、请求头、空行和实体组成。

  • 请求行：包含三个字段：请求方式+请求URL+请求协议版本。
    • 请求方式：GET、POST、HEAD、PUT和DELETE。
    • GET请求：向服务器获取数据。
    • POST请求：将实体提交到服务器指定的资源，通常情况下会修改服务器的数据。
    • HEAD请求：HEAD请求和GET请求类似，但在返回的响应中不包含请求对象。
    • PUT请求：上传文件到指定服务器。
    • DELETE：删除服务器上的内容。

HTTP响应报文

响应报文由：响应行+响应头+响应实体组成。

  • 响应行：包含协议版本+状态码+状态信息组成。
  • 响应实体：包含客户端所请求的对象。

常见的状态码：

  • 200：请求成功
  • 202：服务端已经收到信息，但尚未进行处理。
  • 301：永久重定向（资源已经被永久移动）。
  • 302：临时重定向（资源临时移动）。
  • 304：请求资源尚未修改。

若使用浏览器本地存储，向服务器发送请求时，服务器回返回304，使用浏览器本地存储。

  • 400：客户端请求语法错误。
  • 403：服务器拒绝该请求。
  • 404：请求资源不存在。
  • 500：服务器内部错误。

首部行

首部行分为四种：请求首部、响应首部、通用首部和实体首部。

通用首部在请求报文和响应报文中都可以设置。

常见请求首部：

  • Accept：可接收的资源类型。
  • Accept-Charset：可接收字符集
  • Host：请求主机名

常见响应首部：

  • ETag：资源匹配信息。
  • Location：客户端重定向URI。

常见通用首部：

  • Cache-Control：控制缓存策略。
  • Connection：管理持久连接。

常见实体首部：

  • Content-Length：实体主题大小。
  • Expires：实体主体过期时间
  • Last-Modified：资源最后修改时间。

HTTP1.1缺点：默认使用持久连接，多个请求复用一个TCP连接，数据请求的通信次数是固定的。服务器完成一个请求后，才能处理下一个请求。若前面请求处理的特别慢，则会造成头部阻塞的情况。队头阻塞会导致持久连接达到最大数量时，剩余资源要等待其他资源完成后才能发起请求。

HTTP2

为了解决HTTP历史版本存在问题，HTTP2应运而出。

二进制协议

HTTP2是一个二进制协议。在HTTP1.1中，报文的头部信息必须是文本（ASCII编码），数据可以是文本，也可以二进制。但在HTTP2中头信息和数据实体都是二进制，并且统称为“帧”。

帧的概念是实现其多路复用的基础。

多路复用

HHTP2实现了多路复用，也是基于TCP连接，但在一个连接里面，客户端和服务端都可以同时发送多个请求或响应，不必按照顺序一一发送，因此解决了“对头阻塞”的问题。

数据流

HTTP2使用数据流的概念 ，因为HTTP2数据包并不是按照顺序发送，同一个连接里面可能存在不同的数据包，可能属于不同的请求。

因此，必须要对数据包做标记，指出它属于哪个请求。HTTP/2 将每个请求或回应的所有数据包，称为一个数据流。每个数据流都有一个独一无二的编号。数据包发送的时候，都必须标记数据流 ID ，用来区分它属于哪个数据流。

头部信息压缩

HTTP/2 实现了头信息压缩，由于 HTTP 1.1 协议不带有状态，每次请求都必须附上所有信息。所以，请求的很多字段都是重复的，比如 Cookie 和 User Agent ，一模一样的内容，每次请求都必须附带，这会浪费很多带宽，也影响速度。HTTP/2 对这一点做了优化，引入了头信息压缩机制。一方面，头信息使用 gzip 或 compress 压缩后再发送；另一方面，客户端和服务器同时维护一张头信息表，所有字段都会存入这个表，生成一个索引号，以后就不发送同样字段了，只发送索引号，这样就能提高速度了。

服务器推送

HTTP2允许服务器未经请求向客户端发送资源（静态资源）。
